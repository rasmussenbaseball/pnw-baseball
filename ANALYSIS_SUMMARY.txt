================================================================================

EXECUTIVE SUMMARY
================================================================================

The conference game count imbalance in playoff projections is caused by one 
fundamental issue: INCORRECT OR MISSING is_conference FLAGS in future_schedules.json

Root Cause Chain:
  1. future_schedules.json is currently EMPTY (0 games)
  2. Once scraper populates it, imbalances will occur if is_conference is wrong
  3. is_conference flag determines which games are counted per team
  4. If flagging is wrong, teams in same conference have different game counts

THE CORE RULE
================================================================================

For every conference, this MUST be true for ALL teams:

  (conference_wins + conference_losses) + conference_games_remaining 
    = SAME NUMBER FOR ALL TEAMS

Why? Every conference game involves exactly 2 teams from that conference.
If Team A plays Team B (both in Conference X), both must count it.

If Team A shows 8 remaining games but Team B shows 7, one game is missing
or miscategorized.

DETECTION SYSTEM (3-Tier)
================================================================================

Tier 1: Site-Level Detection (UNRELIABLE)
  - Sidearm: Parse HTML data-is-conference attributes
  - NWAC: Check if both teams in NWAC_TEAMS_SET and same division
  - Problem: Different sites encode this inconsistently

Tier 2: Database Validation (AUTHORITATIVE - CORRECT APPROACH)
  - Check if home_team.conference_id == away_team.conference_id
  - Located in scrape_future_schedules.py lines 884-912
  - OVERWRITES Tier 1 detection

Tier 3: Counting Games (CALCULATION)
  - Sum games where is_conference=true for each team
  - Located in projections.py line 451
  - Uses the (potentially incorrect) is_conference flag from Tier 1 & 2

WHERE IMBALANCE HAPPENS
================================================================================

Imbalance occurs when is_conference flagging is WRONG at database validation 
step. Five common scenarios:

1. GAME NEVER SCRAPED
   - Team's website doesn't list it (or scraper can't parse format)
   - One team counts game, other doesn't
   - Example: "Whitworth @ UPS" appears on Whitworth's site but not UPS's

2. TEAM NOT IN TEAM_MAP
   - Team exists in database but scraper doesn't know about it
   - home_team_id or away_team_id = None
   - Database validation can't match conference_ids
   - Game incorrectly marked as NOT CONFERENCE

3. TEAM HAS WRONG CONFERENCE_ID IN DATABASE
   - Team moved conferences but database not updated
   - Validation finds home_conf != away_conf
   - Game marked as NOT CONFERENCE (even though it should be)
   - Both teams lose count

4. DUPLICATE GAMES WITH DIFFERENT is_conference VALUES
   - Same game appears twice in scraped data
   - One copy says is_conference=true, other says false
   - Deduplication picks wrong copy
   - Teams lose count inconsistently

5. CROSS-CONFERENCE GAME MARKED AS CONFERENCE
   - Exhibition or neutral opponent game incorrectly marked
   - SHOULD be caught by database validation
   - But fails if team_id can't be resolved

QUESTIONS?
================================================================================

Q: Why isn't the scraper automatically run?
A: It requires scraping multiple websites (Sidearm, PrestoSports) which may
   be rate-limited or have WAF protection. Should be added to daily_update.sh.

Q: Can the imbalance be silent (data looks good but is wrong)?
A: Yes! The diagnostic checks for it, but normal viewing won't show it.
   Teams might incorrectly project as making or missing playoffs.

Q: How often should this be rechecked?
A: Every time scraper is updated or websites change their HTML structure.

Q: What if one conference is balanced but another isn't?
A: Different conferences may have different sources (Sidearm vs PrestoSports
   vs custom). Debug the problematic one independently.
